Why use OpenPie?

We believe that there are huge opportunities ahead by solving an annoying problem: devices such as smartphones are advanced pieces of hardware today, but people are still getting frustrated trying to share simple daily informations.

Human communication flows naturally. You don’t need to ask your friend about his “ear’s OS” before delivering an information. When people talk, things are quite simple and fun.

But the same is not true to humans’ devices. To get a full example, please refer to “Why Joe, Amanda and Mike are my friends, but their devices are not”.

What is a Pie

Pie is an acronym of Piece of information. Things we want to share daily are just pieces of info: parties invitations, places to go, recipes, reviews, quotes, business cards...

Introduction to OpenPie

OpenPie adds a layer of freedom and fun to devices.

OpenPie is a simple combination of structured data (JSON) and vocabulary (Schema.org) to make applications talk to each other, regardless: the operating systems involved, device limitations (eg. is there a camera to read visual code or shoud use NFC?) and the applications themselves being the same or not.

It is suitable to preserve the data during data-interchange procedure and, because it uses a vocabulary to describe the information, to give a context to the transmitted data.

It’s also open.

Giving and receiving pies

JSON is the initial approach for transmitting pies because it is familiar to most programmers and completely language independent.

To make sense, the applications involved must be talking the same language, or, in a better definition, the same vocabulary.

The chosen vocabulary is Schema.org. It is public, easy to understand, friendly to search engines and towards semantic web efforts.

Code example

We want to create a pie for Joe’s party. This piece of info may contains:

  • Title: Joe’s Party
  • Description: Let’s rock and bake some pies!
  • Date: November/24/2012
  • Time: 14:50

A party is an event.

Schema.org defines a set of possible properties to describe it. That schema is available at http://schema.org/Event

The resulting pie is a JSON string containing all the information put into suitable properties:

{"itemtype":"http://schema.org/Event","name":"Joe's party","description":"Let's rock and bake some pies!","startDate":"2012-11-24T14:50"}

Just for fun, we can call this code the pie recipe.

Note that itemtype refers to the chosen schema, so it is possible to the other side (the application that is receiving the JSON string) to understand what it is about and manage the data according to the context.
